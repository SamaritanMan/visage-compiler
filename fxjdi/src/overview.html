This documentation describes the extensions and additions that have been made to the Java Debug Interface (JDI) in order to allow debugging of Java FX programs. These extensions and additions are called FX-JDI.

FX-JDI is implemented as a set of classes that wrap the normal JDI implementation classes. These FX-JDI classes implement the JDI interfaces so that debuggers will in general just access the methods defined in the JDI interfaces. Debuggers will only need to access these FX-JDI implementation classes directly in order to use the new classes and methods that have been added and thus are not in the JDI interfaces.

FX-JDI is experimental and subject to change in future releases.

The FX-JDI classes are included in the file fxjdi.jar. In order for a client to use FX-JDI, fxjdi.jar must be on the client's classpath. The client should first get a {@link com.sun.jdi.VirtualMachineManager} object:

com.sun.jdi.VirtualMachineManager vmm = org.visage.jdi.FXBootstrap.virtualMachineManager();

This gives the client a JavaFX enabled VirtualMachineManager (org.visage.jdi.FXVirtualMachineManager). The client can then get one of the following FX-JDI connectors from the VirtualMachineManager:

  • org.visage.jdi.connect.FXLaunchingConnector (this is the default connector)
  • org.visage.jdi.connect.FXProcessAttachingConnector
  • org.visage.jdi.connect.FXRawLaunchingConnector
  • org.visage.jdi.connect.FXSocketAttachingConnector
  • org.visage.jdi.connect.FXSocketListeningConnector
  • org.visage.jdi.connect.FXSharedMemoryAttachingConnector - currently only available on Windows
  • org.visage.jdi.connect.FXSharedMemoryListeningConnector - currently only available on Windows

    • The connector object can be used to obtain an {@link org.visage.jdi.FXVirtualMachine} as described in the JDI documentation.

    This javadoc shows the classes and methods that FX-JDI adds to JDI, as well as the enhancements to existing JDI methods. Due to limitations of javadoc, the existing unmodified methods in these classes are also shown. These methods have no javadoc comments, and behave as described in the JDI spec.
