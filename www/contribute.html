How to Contribute to the OpenJFX Compiler Project

This page describes the contribution process for the OpenJFX Compiler project. This interim process will be used until an open governance model for the project is established.

The design of this process is based loosely upon the contribution process of the OpenJDK project.

0. Become a contributor

Like many other open-source projects, the OpenJFX Compiler project requires contributors to jointly assign their copyright on contributed code. If you haven't yet signed the Sun Contributor Agreement (SCA), then please download and review the SCA itself. Go here for details on remitting the SCA and about its acceptance.

The SCA gives Sun and the contributor joint copyright interests in the code. The contributor retains copyrights while also granting those rights to Sun as the project sponsor. You only need to sign the SCA once in order to cover all changes that you might contribute to any Sun-sponsored open-source project including not just the OpenJFX Compiler project but also, for example, GlassFish, NetBeans, and OpenSolaris. If you've already signed the SCA in order to contribute to some other Sun-sponsored project, then you do not need to sign it again in order to contribute to the OpenJFX Compiler project.

1. Find something interesting to work on

The most obvious thing to work on is a bug or enhancement (RFE) about which you are passionate. Please check the "visage-compiler" component of the online bug database to see if your idea is already described there, and if so then use that bug/RFE's id number when writing about your work.

If you're interested in a particular area but don't have any specific ideas about what to do, then feel free to post a query to the appropriate development mailing list to ask for suggestions that match your skills and knowledge.

2. Submit a test

If you have some Visage source code that does not compile correctly, attempt to trim the file down to the main problem, and attach it to an issue describing the problem. We'll add that file to our test suite.

3. Discuss your intended change

Before you invest time working on a change, discuss what you're planning to do with other engineers working on the same code. They're likely to be able to offer comments and suggestions that will result in a higher-quality change and a smoother submission process. Announcing that you're working on a particular item can also help to avoid wasted effort in case someone else is already working on it.

If you're thinking of working on an existing bug or RFE, then the best way to start such a discussion is to post a message to the appropriate development mailing list with a subject line of the form

IssueID 210: My favorite bug

where 210 is replaced with the actual bug or RFE id number and My favorite bug is replaced with the bug or RFE's synopsis.

4. Submit a patch

When your change is ready, send a message to the appropriate development mailing list with a subject line of the form

[PATCH] 210: My favorite bug

where 210 and My favorite bug are replaced as above. If no existing bug or RFE describes your change then omit the id number and one will be created and assigned.

The message should contain the following two things, either directly in the body or as attachments:

  • A description of the change, including a rationale for your approach, and a discussion of any alternative approaches that were considered, if relevant. If there is no existing bug or RFE already describes your change, then also include a description of the problem that it solves or the need that it addresses.
  • Diffs of the code changes, in unified diff format (-u). Be sure to identify the baseline version for your diffs by mentioning its build number (b02, b03, etc.), which you can find in the name of the source bundle you that started with. If you're using a working copy of the Subversion repository then you can find the build number in the output of the svn log command, and diffs in the required format can be generated via the svn diff command.

5. Work with your sponsor

A Sun engineer whose knowledge and skills are a good match for your proposed change will be assigned as your sponsor and will reply to your message.

You can generally expect a reply within about five working days. Actual response times will vary due to factors, such as impending product-release deadlines, contribution volume, and holiday schedules. You should also expect a longer response time if you haven't already discussed your proposed change on the appropriate development list, since it will take more time for your sponsor to come up to speed on what you're trying to do.

Your sponsor will first evaluate your patch for correctness, compatibility, stability, performance, coding style, and overall appropriateness to the current phase of development. If your change passes inspection then your sponsor will work with you to address any minor deficiencies and then guide you through the rest of the development process, navigating the relevant internal Sun engineering processes on your behalf and ultimately, if all goes well, integrating your patch into the JDK.

6. Know what to expect

Only the best patches submitted will actually make it all the way into the Visage compiler source base. Sun's goal is not to accept the maximum number of contributions possible, but rather to accept only high-quality contributions that improve the Visage compiler.

If you're relatively new to the Java platform and the Visage language, then we recommend that you gain more experience writing Java and Visage applications before you attempt to work on the Visage Compiler itself. Sun's aim here is to bring developers who already have the skills required to work on the Visage compiler into the existing OpenJFX development community. The members of that community have neither the time nor the patience required to teach basic Java programming skills or platform implementation techniques. (Sun offers formal Java language training if that's what you're looking for.) You can learn the current Visage language implementation by visiting the OpenJFX web site.
